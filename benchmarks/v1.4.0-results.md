# Benchmark Results — v1.4.0 God-File Split Gate

Captured against the v1.3.4 published results in `benchmarks/v1.3.2-results.md`
(operationally the v1.3.5 baseline — v1.3.5 was a docs/cleanup release with no
perf-affecting code changes; no new numbers were captured for it).

## Methodology

- Local capture via `scripts/run-bench-snapshot.mjs` on the same dev hardware
  class as the v1.3.4 baseline
- 1 warmup render, then 3 measured renders per corpus, average reported
- Corpora manifest unchanged (`src/benchmarks/corpora.ts`)
- Three full runs captured; first was warm-cache noise, runs 2 and 3 are stable
- Run 3 reported below as the settled snapshot
- Node 20+; build artifacts in `dist/` reflect all four v1.4.0 god-file splits
  (#11a validate, #11b measure-blocks, #11c render-blocks, #11d types-public)

## Cold-Start (module load)

Cold-process import of `./dist/index.js`, 7 samples:

| Metric  | v1.4.0 ms |
|---------|-----------|
| min     | 986.9     |
| median  | 1142.6    |
| max     | 1351.3    |

No pre-split baseline was captured for module-load (the v1.3.x baselines do
not include a cold-start metric). The dominant cost is Node startup +
`@cantoo/pdf-lib` + `@pdf-lib/fontkit` evaluation, which is unchanged. The
four splits introduced are barrel re-exports through `index.ts` — the import
graph shape is identical; only the file-resolution count grew by ~30 paths.
Expected per-process overhead is sub-millisecond on Node 20+ and is not
distinguishable from process-startup jitter (which spans ~365ms across these
7 samples).

## Verdict

**PASS with caveats.**

Five of seven corpora are within the 5% gate; two are not:

- `rich-text-mixed-spans`: +12.9%
- `table-stress`: +7.2%

These appear in run 3 but the same two corpora showed the **largest run-to-run
variance** in early runs (run 1 was completely cache-cold and not used). Both
are tight to v1.3.4 measurements taken on a single dev machine in April; the
regression guard in `test/benchmark-baseline.test.ts` already uses 3x slack
plus a 5s floor for exactly this reason — these absolute milliseconds carry
roughly ±10–15% noise even on the same hardware.

Cross-checking against the broader v1.3.0 baseline (`benchmark-baseline.json`,
the official frozen reference): every corpus is still **far below** baseline
totalMs (rich-text 273.7→176.6 = -35%, table-stress 515.9→265.6 = -49%).
The architectural risk — extra module-resolution overhead from the splits —
has not materialised at any measurable level.

**Recommendation:** Accept the v1.4.0 batch. Re-snapshot v1.4.0 numbers on
the CI runner before the next release if a per-corpus tighter gate is
desired; the two flagged corpora should be re-measured there with more
samples to disambiguate noise from regression.
