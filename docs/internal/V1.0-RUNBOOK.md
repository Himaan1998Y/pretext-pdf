## Part 1 — Mental Model Audit

This runbook was produced after explicit application of the following lenses. Each lens changed at least one decision in the v1 draft.

### 1.1 First Principles

What does v1.0 functionally require, with nothing assumed?

| Requirement | State |
|---|---|
| Code does what it claims | 668 tests pass; build green |
| Public API stable for v1.x | Mostly stable; plugin types remain `@beta` |
| Documentation matches code | Gaps (README has zero plugin mentions, CHANGELOG stops at 0.9.3) |
| Release artifacts contain promised features | Gap (plugin files absent from published 0.9.3 dist) |
| No crashes on supported platforms | Gap (no upfront `Intl.Segmenter` guard) |

What is NOT required but was over-scoped in v1?
- Calibrated benchmark regression detection (1.0.1)
- JSDoc on every undocumented public symbol (most are self-explanatory)
- Plugin support in every compound type (only `float-group` accepts element children)
- Plugin registration tool in MCP (security risk; do not ship)

### 1.2 Inversion

What would make v1.0 a disaster, and how do we prevent it?

| Failure | Mitigation |
|---|---|
| npm publishes broken bundle | Phase 5 dist-verify gate |
| Plugin API ships then changes in v1.1 | `@beta` marker preserved |
| H-7 implemented incompletely | Scope strictly to float-group, the only host |
| MCP enables RCE via plugin tool | Decision: do not ship plugin tool |
| Tag pushed before final review | Manual confirmation gate before push |
| Hardware-specific benchmark fails on different machine | Calibration deferred to 1.0.1 |

### 1.3 Pareto / 80–20

The high-value 20%: Phases 0, 2, 3, 5 ship a real documented v1.0.
The lower-value 80%: H-7 (Phase 1), benchmarks (Phase 4), MCP polish (Phase 6), cleanup (Phase 7) — defensible to defer.

**Decision:** H-7 becomes Phase 1-OPTIONAL. The runbook includes both paths (slim v1.0 / full v1.0).

### 1.4 Cynefin

| Domain | Items | Approach |
|---|---|---|
| Clear | TocEntryElement export, version bump, CHANGELOG | Just do |
| Complicated | H-7 implementation | Apply expertise (architect + TDD) |
| Complex | MCP plugin tool design | Decide via threat-model (won't ship) |
| Chaotic | "pretext fork" confusion | Investigated first, then categorized |

### 1.5 Reversibility

One-way doors that need extra care:
1. `npm publish` of v1.0.0 (cannot unpublish after 72h)
2. Tag push (deletable, but if CI publishes before deletion, see #1)
3. Force push to main (forbidden in this runbook)

Two-way doors (low ceremony):
- All file edits before commit
- All commits before tag
- All draft CHANGELOG entries
- Local builds and benchmarks

### 1.6 Karpathy Guidelines (from CLAUDE.md)

| Guideline | Application |
|---|---|
| Think Before Coding | This runbook IS the thinking |
| Simplicity First | Drop H-7 from minimum-viable v1.0 |
| Surgical Changes | Each phase touches a bounded set of files |
| Goal-Driven Execution | Each step has explicit verify gates |

## Part 2 — Anti-Hallucination Protocol

Every concrete reference in this runbook is one of:

- **VERIFIED**: confirmed in this session via Read, Grep, or Bash
- **TO-VERIFY**: must be re-checked at the moment of use (commands listed)
- **ASSUMED**: stated assumption, with fallback if wrong

Rules for execution:

1. **Before any Edit, Read the file** — even if you "remember" the content
2. **Before any Bash command involving paths, verify the path exists**
3. **Before invoking an agent name, confirm it appears in the project's available agents list**
4. **Before invoking a slash command, confirm it appears in the project's `commands/` directory**
5. **If a referenced npm script is missing from package.json, STOP and re-read package.json**
6. **If an agent's report contradicts a verified fact, trust the verified fact**
7. **Effort estimates in this document are guidance, not commitments — adjust on the ground**

## Part 6 — Parallel Debug Playbook

When a step fails, do not flail. Spawn multiple agents in parallel. Each gets a different angle. Consolidate findings before fixing.

### 6.1 Build failure
1. 🤖 `build-error-resolver` — primary; minimum-diff fix
2. 🤖 `code-reviewer` — review the most recent commit that touched compiler-relevant config
3. 🤖 `Explore` — list all imports of the failing symbol/module

### 6.2 Test failure
1. 🤖 `code-reviewer` — what changed
2. 🤖 `silent-failure-hunter` — was something swallowed
3. 🤖 `pr-test-analyzer` — is the test itself correct

### 6.3 Public API surface drift
1. 🤖 `typescript-reviewer` — type correctness of new exports
2. 🤖 `type-design-analyzer` — invariants and encapsulation
3. 🤖 `Explore` — what files reference the changed symbol

### 6.4 Performance regression
1. 🤖 `performance-optimizer` — find the bottleneck
2. 🤖 `code-reviewer` — recent changes in hot paths
3. 🤖 `code-explorer` — trace the slow path end-to-end

### 6.5 Security concern
1. 🤖 `security-reviewer` — primary threat model
2. 🤖 `silent-failure-hunter` — error paths leak info
3. 🤖 `code-reviewer` — secondary correctness

### 6.6 Documentation accuracy
1. 🤖 `doc-updater` — what's stale
2. 🤖 `comment-analyzer` — JSDoc rot risk
3. 🤖 `Explore` — find every reference to the stale fact
